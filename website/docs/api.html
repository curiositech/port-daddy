Port Management

9 endpoints
POST /claim

Claim a stable port for a service identity. Re-claiming an existing identity returns the same port (idempotent). Port assignment is atomic — no two services will receive the same port.

Request Body

{
  "id": "myapp:api",          // required — semantic identity
  "port": 3000,             // optional — request a specific port
  "range": [3000, 3999],   // optional — restrict to port range
  "expires": 3600000,      // optional — TTL in ms
  "cmd": "npm start",      // optional — start command for orchestration
  "cwd": "/path/to/app"   // optional — working directory
}

Response

{
  "success": true,
  "id": "myapp:api",
  "port": 3001,
  "assigned": true       // false if re-claim of existing port
}

curl Example

$ curl -s -X POST http://localhost:9876/claim \
    -H "Content-Type: application/json" \
    -d '{"id": "myapp:api"}'
DELETE /release

Release a port claim. Releasing a non-existent service returns 200 with released: 0 (idempotent).

Request Body

{
  "id": "myapp:api"    // required
}

Response

{
  "success": true,
  "id": "myapp:api",
  "released": 1       // 0 if not found
}

curl Example

$ curl -s -X DELETE http://localhost:9876/release \
    -H "Content-Type: application/json" \
    -d '{"id": "myapp:api"}'

List all registered services. Optionally filter by project or stack using the project and stack query parameters.

Release stale port claims — services that have not sent a heartbeat within their TTL window.

  "ids": ["old-service:api", "stale:worker"]
}

curl Example

$ curl -s -X POST http://localhost:9876/ports/cleanup
GET /wait/:id

Wait (long-poll) until a service becomes available. Useful for startup sequencing — blocks until the service registers its port claim.

Query Parameters

ParamDescription
timeoutMax wait in ms (default: 30000) optional

Response

{
  "id": "myapp:api",
  "port": 3001,
  "waited": 1240    // ms waited
}

curl Example

$ curl -s "http://localhost:9876/wait/myapp:api?timeout=10000"
POST /wait

Wait for multiple services to become available. Returns when all listed services have claimed a port or the timeout expires.

Request Body

{
  "ids": ["myapp:api", "myapp:db"],
  "timeout": 30000
}

Response

{
  "all": true,
  "services": [
    { "id": "myapp:api", "port": 3001, "ready": true },
    { "id": "myapp:db", "port": 5432, "ready": true }
  ]
}

curl Example

$ curl -s -X POST http://localhost:9876/wait \
    -H "Content-Type: application/json" \
    -d '{"ids": ["myapp:api", "myapp:db"]}'

Sessions & Notes

12 endpoints
POST /sessions

Start a new coordination session. Sessions are the primary unit of multi-agent work tracking. Notes added to a session form an immutable audit trail.

Request Body

{
  "purpose": "Building auth system",  // required
  "agentId": "agent-abc123",          // optional
  "identity": "myapp:api",            // optional — semantic identity
  "type": "work",                    // optional — work | review | debug
  "phase": "planning"               // optional — initial phase label
}

Response

{
  "id": "sess-d4e5f6",
  "purpose": "Building auth system",
  "status": "active",
  "createdAt": "2026-03-10T12:00:00Z"
}

curl Example

$ curl -s -X POST http://localhost:9876/sessions \
    -H "Content-Type: application/json" \
    -d '{"purpose": "Building auth system"}'
GET /sessions

List sessions, optionally filtered by status or agent.

Query Parameters

ParamDescription
statusactive | completed | abandoned optional
agentFilter by agent ID optional
limitMax results (default: 50) optional

curl Example

$ curl -s "http://localhost:9876/sessions?status=active"
GET /sessions/:id

Get full session details including notes and file claims.

curl Example

$ curl -s http://localhost:9876/sessions/sess-d4e5f6
PUT /sessions/:id

Update a session — mark it completed, abandoned, or update metadata.

Request Body

{
  "status": "completed",   // completed | abandoned | active
  "phase": "done"
}

curl Example

$ curl -s -X PUT http://localhost:9876/sessions/sess-d4e5f6 \
    -H "Content-Type: application/json" \
    -d '{"status": "completed"}'
DELETE /sessions/:id

Delete a session and cascade-delete its notes and file claims.

curl Example

$ curl -s -X DELETE http://localhost:9876/sessions/sess-d4e5f6
PUT /sessions/:id/phase

Advance a session to a new named phase. Phases are free-form strings — common values are planning, implementation, review, done.

Request Body

{
  "phase": "implementation"
}

curl Example

$ curl -s -X PUT http://localhost:9876/sessions/sess-d4e5f6/phase \
    -H "Content-Type: application/json" \
    -d '{"phase": "implementation"}'
POST /sessions/:id/notes

Add an immutable note to a session. Notes cannot be edited or deleted individually — they form a permanent audit trail.

Request Body

{
  "content": "Auth middleware implemented, tests passing",
  "type": "progress"    // progress | decision | blocker | handoff
}

curl Example

$ curl -s -X POST http://localhost:9876/sessions/sess-d4e5f6/notes \
    -H "Content-Type: application/json" \
    -d '{"content": "Auth middleware done", "type": "progress"}'
GET /sessions/:id/notes

Get all notes for a session in chronological order.

curl Example

$ curl -s http://localhost:9876/sessions/sess-d4e5f6/notes
POST /sessions/:id/files

Claim files for this session. File claims are advisory — they provide conflict detection warnings but are not enforced locks.

Request Body

{
  "files": ["src/auth.ts", "src/middleware.ts"]
}

Response

{
  "claimed": ["src/auth.ts", "src/middleware.ts"],
  "conflicts": []    // files claimed by another active session
}

curl Example

$ curl -s -X POST http://localhost:9876/sessions/sess-d4e5f6/files \
    -H "Content-Type: application/json" \
    -d '{"files": ["src/auth.ts"]}'
DELETE /sessions/:id/files

Release file claims for this session.

Request Body

{
  "files": ["src/auth.ts"]    // omit to release all files
}

curl Example

$ curl -s -X DELETE http://localhost:9876/sessions/sess-d4e5f6/files \
    -H "Content-Type: application/json" \
    -d '{"files": ["src/auth.ts"]}'

Agents

8 endpoints
POST /agents

Register an agent. The daemon tracks heartbeats — agents that stop heartbeating become stale (10 min) and then dead (20 min), entering the salvage queue.

Request Body

{
  "id": "agent-abc123",         // required
  "identity": "myapp:api",     // optional — project:stack:context
  "purpose": "Building auth", // optional — human-readable task
  "worktreeId": "wt-xyz"      // optional — git worktree ID
}

Response

{
  "id": "agent-abc123",
  "registered": true,
  "salvageHint": {           // present if dead agents in same project
    "count": 2,
    "project": "myapp"
  }
}

curl Example

$ curl -s -X POST http://localhost:9876/agents \
    -H "Content-Type: application/json" \
    -d '{"id": "agent-abc123", "identity": "myapp:api", "purpose": "Building auth"}'
POST /agents/:id/heartbeat

Send a heartbeat to keep the agent alive. Recommended every 5 minutes. Missing heartbeats for 10 minutes marks the agent stale; 20 minutes marks it dead.

Response

{
  "id": "agent-abc123",
  "alive": true,
  "lastHeartbeat": "2026-03-10T12:05:00Z"
}

curl Example

$ curl -s -X POST http://localhost:9876/agents/agent-abc123/heartbeat
DELETE /agents/:id

Unregister an agent. Call this when your agent finishes its work.

curl Example

$ curl -s -X DELETE http://localhost:9876/agents/agent-abc123
GET /agents/:id

curl Example

Locks

5 endpoints
POST /locks/:name

Acquire a named distributed lock. Returns 409 if the lock is already held. Locks have a TTL and auto-expire if not extended.

Request Body

{
  "holder": "agent-abc123",   // required — who holds the lock
  "ttl": 300000             // optional — TTL in ms (default: 300000 = 5 min)
}

Response (200 — acquired)

{
  "acquired": true,
  "name": "db-migrations",
  "holder": "agent-abc123",
  "expiresAt": "2026-03-10T12:10:00Z"
}

Response (409 — already held)

{
  "acquired": false,
  "holder": "agent-xyz789",
  "expiresAt": "2026-03-10T12:08:00Z"
}

curl Example

$ curl -s -X POST http://localhost:9876/locks/db-migrations \
    -H "Content-Type: application/json" \
    -d '{"holder": "agent-abc123"}'
PUT /locks/:name

Extend a lock's TTL. Only the current holder can extend. Returns 403 if called by a different holder.

Request Body

{
  "holder": "agent-abc123",
  "ttl": 300000
}

curl Example

$ curl -s -X PUT http://localhost:9876/locks/db-migrations \
    -H "Content-Type: application/json" \
    -d '{"holder": "agent-abc123", "ttl": 300000}'
DELETE /locks/:name

Release a lock. Only the holder can release. Pass the holder in the request body.

Request Body

{
  "holder": "agent-abc123"
}

curl Example

$ curl -s -X DELETE http://localhost:9876/locks/db-migrations \
    -H "Content-Type: application/json" \
    -d '{"holder": "agent-abc123"}'
GET /locks/:name

Check the current status of a named lock.

curl Example

$ curl -s http://localhost:9876/locks/db-migrations
GET /locks

List all currently held locks.

curl Example

$ curl -s http://localhost:9876/locks

Messaging

6 endpoints
POST /msg/:channel

Publish a message to a channel. Messages are stored and can be retrieved later. Delivery also broadcasts to SSE subscribers in real time.

Request Body

{
  "content": "Build complete",
  "metadata": { "buildId": "build-42" }   // optional
}

curl Example

$ curl -s -X POST http://localhost:9876/msg/builds \
    -H "Content-Type: application/json" \
    -d '{"content": "Build complete"}'
GET /msg/:channel

Get messages from a channel. Optionally paginate with since and limit.

Query Parameters

ParamDescription
limitMax messages to return (default: 50) optional
sinceISO timestamp — only return messages after this time optional

curl Example

$ curl -s "http://localhost:9876/msg/builds?limit=10"
DELETE /msg/:channel

Clear all messages from a channel.

curl Example

$ curl -s -X DELETE http://localhost:9876/msg/builds
GET /msg/:channel/poll

Long-poll for new messages. Blocks until a new message arrives or the timeout expires. Use for lightweight event-driven coordination without SSE.

Query Parameters

ParamDescription
sinceISO timestamp — wait for messages after this time optional
timeoutMax wait in ms (default: 30000) optional

curl Example

$ curl -s "http://localhost:9876/msg/builds/poll?timeout=10000"
SSE /msg/:channel/subscribe Server-Sent Events

Subscribe to a channel via Server-Sent Events. Keeps a persistent connection open and pushes new messages as they arrive. Max 10 concurrent SSE connections per IP.

Event Format

data: {"id": "msg-001", "content": "Build complete", "timestamp": "2026-03-10T12:00:00Z"}

data: {"id": "msg-002", "content": "Tests passed"}

JavaScript Example

const es = new EventSource('http://localhost:9876/msg/builds/subscribe');
es.onmessage = (e) => console.log(JSON.parse(e.data));
GET /channels

List all pub/sub channels that have messages or active subscribers.

curl Example

$ curl -s http://localhost:9876/channels

Salvage

6 endpoints

When an agent dies (heartbeat timeout), its work enters the salvage queue. Another agent can claim the dead agent's context and continue the work. The deprecated /resurrection/* aliases still work but point to these routes.

GET /salvage

List all entries in the salvage queue — dead agents whose work can be claimed.

Query Parameters

ParamDescription
projectFilter by project prefix optional
stackFilter by stack segment optional

Response

{
  "queue": [
    {
      "agentId": "agent-old123",
      "identity": "myapp:api",
      "purpose": "Building auth",
      "diedAt": "2026-03-10T11:00:00Z",
      "status": "pending"
    }
  ]
}

curl Example

$ curl -s "http://localhost:9876/salvage?project=myapp"
GET /salvage/pending

List agents actively pending salvage (not yet claimed, not yet dismissed).

curl Example

$ curl -s http://localhost:9876/salvage/pending
POST /salvage/claim/:agentId

Claim a dead agent's work. Returns the dead agent's sessions and notes so you can continue where it left off.

Request Body

{
  "rescuerId": "agent-new456"   // optional — your agent ID
}

Response

{
  "claimed": true,
  "deadAgent": {
    "id": "agent-old123",
    "purpose": "Building auth",
    "sessions": [{ "id": "sess-xyz", "notes": [...] }]
  }
}

curl Example

$ curl -s -X POST http://localhost:9876/salvage/claim/agent-old123 \
    -H "Content-Type: application/json" \
    -d '{"rescuerId": "agent-new456"}'
POST /salvage/complete/:agentId

Mark salvage complete — the dead agent's work has been finished. Removes the entry from the queue.

curl Example

$ curl -s -X POST http://localhost:9876/salvage/complete/agent-old123
POST /salvage/abandon/:agentId

Return a claimed agent back to the salvage queue for another agent to pick up.

curl Example

$ curl -s -X POST http://localhost:9876/salvage/abandon/agent-old123
DELETE /salvage/:agentId

Dismiss a dead agent from the salvage queue without claiming its work.

curl Example

$ curl -s -X DELETE http://localhost:9876/salvage/agent-old123

Sugar

3 endpoints

Sugar endpoints compose multiple operations into single atomic calls. They are the recommended way to start and end agent work sessions — equivalent to pd begin and pd done in the CLI.

POST /sugar/begin

Register an agent and start a session atomically. Equivalent to calling POST /agents then POST /sessions.

Request Body

{
  "identity": "myapp:api",          // required
  "purpose": "Building auth system", // required
  "agentId": "agent-abc123"         // optional — auto-generated if omitted
}

Response

{
  "agentId": "agent-abc123",
  "sessionId": "sess-d4e5f6",
  "identity": "myapp:api",
  "salvageHint": null    // non-null if dead agents found in project
}

curl Example

$ curl -s -X POST http://localhost:9876/sugar/begin \
    -H "Content-Type: application/json" \
    -d '{"identity": "myapp:api", "purpose": "Building auth system"}'
POST /sugar/done

End the session and unregister the agent atomically. Adds a closing note if provided. Falls back to finding any active session if agentId is not specified.

Request Body

{
  "agentId": "agent-abc123",   // optional — use context if omitted
  "note": "Auth complete, tests passing"  // optional closing note
}

Response

{
  "agentId": "agent-abc123",
  "sessionId": "sess-d4e5f6",
  "ended": true
}

curl Example

$ curl -s -X POST http://localhost:9876/sugar/done \
    -H "Content-Type: application/json" \
    -d '{"agentId": "agent-abc123", "note": "Auth complete"}'
GET /sugar/whoami

Get the current agent and session context. Reads from .portdaddy/current.json in the working directory if present.

Response

{
  "agentId": "agent-abc123",
  "sessionId": "sess-d4e5f6",
  "identity": "myapp:api",
  "purpose": "Building auth system",
  "files": ["src/auth.ts"]
}

curl Example

$ curl -s http://localhost:9876/sugar/whoami
